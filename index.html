.gitignore

The .gitignore file is a fundamental component in Git for managing version control by specifying intentionally untracked files and directories that Git should ignore. It prevents these specified items from being included in the repository's history and from appearing as "untracked files" in git status.

Daily Workflow

Status & Diff

  • git status ‚Äî Check working directory status
  • git diff ‚Äî Show unstaged changes
  • git diff --staged ‚Äî Show staged changes
  • git diff <file_name> ‚Äî Show changes in specific file

Git Log

  • git log ‚Äî Show commit history
  • git log --oneline ‚Äî Compact one-line view
  • git log --graph ‚Äî Show branch structure
  • git log -n 5 ‚Äî Show last 5 commits
  • git log --author="name" ‚Äî Filter by author
  • git log --grep="keyword" ‚Äî Search commit messages

git log displays your repository's commit history with details like author, date, and messages. Essential for tracking changes and understanding what happened in your project over time.

Git Stash (Temporary Storage)

  • git stash ‚Äî Save changes temporarily
  • git stash save "message" ‚Äî Stash with a description
  • git stash list ‚Äî List all stashes
  • git stash pop ‚Äî Apply and remove latest stash
  • git stash apply ‚Äî Apply latest stash without removing it
  • git stash apply stash@{n} ‚Äî Apply specific stash
  • git stash drop stash@{n} ‚Äî Delete specific stash
  • git stash clear ‚Äî Remove all stashes

git stash is a Git command that temporarily saves your uncommitted changes (both staged and unstaged) so you can work on something else, then come back and re-apply them later. It's useful when you need to switch branches quickly but aren't ready to commit your current work. Think of it as a clipboard for your changes.

Branching & Merging

Git Branches

  • git branch ‚Äî List all local branches
  • git branch <branch_name> ‚Äî Create a new branch
  • git branch -d <branch_name> ‚Äî Delete a branch (safe delete)
  • git branch -D <branch_name> ‚Äî Force delete a branch
  • git checkout <branch_name> ‚Äî Switch to an existing branch
  • git checkout -b <branch_name> ‚Äî Create and switch to a new branch
  • git switch <branch_name> ‚Äî Modern way to switch branches
  • git switch -c <branch_name> ‚Äî Create and switch to a new branch
  • git merge <branch_name> ‚Äî Merge specified branch into current branch
  • git branch -M <new_name> ‚Äî Rename current branch

Git branches are lightweight pointers to commits that allow you to work on different features, bug fixes, or experiments in isolation. They enable parallel development where multiple people can work on different parts of a project simultaneously without interfering with each other's work. Branches are a core feature of Git's distributed version control system.

Pull Requests (PR)

  • Step 1: Create a new branch ‚Äî git checkout -b feature-branch
  • Step 2: Make changes and commit ‚Äî git add . && git commit -m "message"
  • Step 3: Push branch to remote ‚Äî git push origin feature-branch
  • Step 4: Create PR on GitHub/GitLab ‚Äî Go to repository and click "New Pull Request"
  • Step 5: Review and discuss changes ‚Äî Team reviews code and suggests improvements
  • Step 6: Merge PR ‚Äî Click "Merge Pull Request" after approval
  • Step 7: Delete branch (optional) ‚Äî git branch -d feature-branch

A Pull Request (PR) is a method of submitting contributions to a project. It lets you tell others about changes you've pushed to a branch in a repository. Once a pull request is opened, you can discuss and review the potential changes with collaborators and add follow-up commits before merging into the main branch. PRs are essential for code review, collaboration, and maintaining code quality in team projects.

Merge Conflicts

  • git status ‚Äî Check which files have conflicts
  • Open conflicted files and look for conflict markers: <<<<<<<, =======, >>>>>>>
  • Edit files to resolve conflicts manually
  • git add <resolved-file> ‚Äî Mark conflict as resolved
  • git commit ‚Äî Complete the merge
  • git merge --abort ‚Äî Abort merge and go back to previous state
  • git mergetool ‚Äî Use visual merge tool to resolve conflicts

Merge conflicts occur when Git can't automatically resolve differences between two commits. This happens when two branches modify the same part of a file differently. Git will mark the conflicts in the file and require you to manually choose which changes to keep. Resolving conflicts is a normal part of collaborative development and helps ensure code quality and consistency.

Remote Repositories

Git Remote

  • git remote -v ‚Äî List all remotes
  • git remote add <name> <url> ‚Äî Add remote repository
  • git remote remove <name> ‚Äî Remove remote
  • git remote rename <old> <new> ‚Äî Rename remote
  • git remote set-url <name> <new-url> ‚Äî Change remote URL

git remote manages connections to remote repositories (like GitHub). Origin is the default remote name, but you can add multiple remotes to collaborate with different repositories or teams.

Git Clone

  • git clone <repository_url> ‚Äî Clone a repository into a new directory
  • git clone <url> <directory_name> ‚Äî Clone into a specific directory
  • git clone -b <branch_name> <url> ‚Äî Clone a specific branch
  • git clone --depth 1 <url> ‚Äî Shallow clone (only latest commit)

git clone is a Git command used to create a copy of an existing repository. It downloads the entire repository including all files, branches, and commit history to your local machine. This is typically the first command you run when you want to work on an existing project from GitHub, GitLab, or other remote repositories.

Git Fetch

  • git fetch ‚Äî Download objects and refs from remote
  • git fetch origin ‚Äî Fetch from origin remote
  • git fetch --all ‚Äî Fetch from all remotes
  • git fetch --prune ‚Äî Remove deleted remote branches

git fetch is a Git command that downloads commits, files, and references from a remote repository into your local repository, but doesn't merge them. Unlike git pull, fetch lets you review changes before integrating them into your work. It's a safer way to see what others have committed without affecting your current work.

Git Pull

  • git pull ‚Äî Fetch and merge changes from remote
  • git pull origin <branch_name> ‚Äî Pull from specific branch
  • git pull --rebase ‚Äî Pull and rebase instead of merge
  • git pull --no-commit ‚Äî Pull without auto-committing merge

git pull is a Git command that fetches changes from a remote repository and immediately merges them into your current branch. It's a combination of git fetch followed by git merge. Use this command to update your local branch with the latest changes from the remote repository.

Advanced Git

Git Rebase

  • git rebase <branch> ‚Äî Rebase current branch onto another branch
  • git rebase -i HEAD~n ‚Äî Interactive rebase (edit last n commits)
  • git pull --rebase ‚Äî Pull with rebase instead of merge
  • git rebase --abort ‚Äî Cancel rebase and go back to before
  • git rebase --continue ‚Äî Continue after resolving conflicts

git rebase moves commits from one branch onto another, creating a linear history. Unlike merge, it doesn't create a merge commit. Use it to keep your branch history clean and organized. Be careful when rebasing shared branches!

Undo Changes

Git Restore

  • git restore [options] <file> ‚Äî Restore file in working directory
  • git restore --staged <file> ‚Äî Unstage file
  • git restore --source <commit_hash> <file> ‚Äî Restore from specific commit

git restore is a Git command used to undo changes in your working directory or staging area (index). It helps you restore files to a previous state ‚Äî like undoing edits or un-staging files.

Git Reset

  • git reset <commit_hash> ‚Äî Move HEAD to specified commit (default: --mixed mode)
  • git reset --soft <commit_hash> ‚Äî Keep changes in staging area
  • git reset --mixed <commit_hash> ‚Äî Keep changes in working directory but unstage them
  • git reset --hard <commit_hash> ‚Äî Discard all changes completely
  • git reset HEAD~1 ‚Äî Undo the last commit

git reset is a Git command used to undo commits by moving the current branch pointer to a different commit. Unlike git revert, it rewrites commit history by removing commits. It's powerful but should be used carefully, especially on shared branches, as it can cause issues for collaborators.

Git Revert

  • git revert <commit_hash> ‚Äî Create new commit that reverses target commit
  • git revert HEAD ‚Äî Undo the last commit
  • git log ‚Äî Find the commit hash first

git revert is a Git command used to undo a commit by creating a new commit that reverses the changes of a previous commit. Unlike git reset, it does not remove history; it safely undoes changes without rewriting commit history.

Git Amend

  • git commit --amend ‚Äî Modify the last commit
  • git commit --amend -m "new message" ‚Äî Change the last commit message
  • git commit --amend --no-edit ‚Äî Add changes to last commit without changing message
  • git add <file> && git commit --amend --no-edit ‚Äî Add forgotten files to last commit
  • git commit --amend --author="Name <email@example.com>" ‚Äî Change commit author

git commit --amend is a Git command used to modify the most recent commit. It allows you to change the commit message, add forgotten files, or fix small mistakes without creating a new commit. This command rewrites history, so avoid using it on commits that have already been pushed to a shared repository, unless you're working on a private branch.

‚ö° GitHub Actions

Automate your workflows! Build, test, and deploy code automatically on events like push or pull requests. Define workflows in YAML files at .github/workflows.

Key Concepts

  • Workflow: YAML file that defines your automation
  • Event: What triggers the workflow (push, PR, schedule, etc.)
  • Job: Collection of steps that run on the same runner
  • Step: Individual command or action to execute
  • Runner: Server that executes your workflow

Common Triggers

  • push ‚Äî When code is pushed
  • pull_request ‚Äî When PR is created/updated
  • schedule ‚Äî On a cron schedule
  • workflow_dispatch ‚Äî Manual trigger

Execution Process

  1. Event Triggered ‚Äî Something happens (push, PR, schedule, etc.)
  2. Workflow Starts ‚Äî GitHub reads your YAML file
  3. Jobs Run ‚Äî Each job executes on a separate runner (parallel by default)
  4. Steps Execute ‚Äî Actions run sequentially within each job
  5. Results ‚Äî Success ‚úÖ or failure ‚ùå ‚Äî you get notified

Cron Schedule

Run workflows automatically on a schedule using cron syntax:

on:
  schedule:
    - cron: '0 0 * * *'  # Every day at midnight UTC
    - cron: '*/15 * * * *'  # Every 15 minutes
    - cron: '0 8 * * 1'  # Every Monday at 8 AM UTC

Cron Format: minute hour day month weekday (5 fields)

Concurrency

Control how many workflow runs execute simultaneously:

name: CI
on: [push]
concurrency:
  group: ${{ github.ref }}
  cancel-in-progress: true  # Cancel previous runs

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
  • group: Run classification (same ref = same group)
  • cancel-in-progress: Cancel old runs when a new one starts

Caching

Speed up workflows by caching dependencies and build outputs:

Method 1: Using actions/cache (Manual)

- uses: actions/cache@v3
  with:
    path: node_modules
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}

Full control over caching logic ‚Äî you specify what to cache

Method 2: Built-in setup-node caching (Recommended)

- uses: actions/setup-node@v3
  with:
    node-version: '18'
    cache: 'npm'  # Auto-caches node_modules based on package-lock.json

Simpler and automatically handled ‚Äî setup-node does the caching for you!

Artifacts

Store files created during workflow runs for later use:

  • Upload: Save build outputs, logs, or reports
  • Download: Retrieve in other jobs or after workflow completion
  • Retention: Stored for 90 days (configurable)
- uses: actions/upload-artifact@v3
  with:
    name: build-files
    path: build/
- uses: actions/download-artifact@v3
  with:
    name: build-files
